1. Principios de diseño — versión final

— Supabase es la única fuente de verdad
Los agentes solo leen y escriben en Supabase. Notion muestra esos datos. Si Notion desaparece, el sistema sigue funcionando sin cambios.

— Google Drive es repositorio, no fuente de datos en tiempo real
El equipo trabaja en Drive como siempre. Los assets se registran explícitamente en Supabase cuando son relevantes para el sistema. Los agentes consultan Supabase, no Drive directamente, excepto para fetch puntual de contenido de un archivo específico.

— Sync unidireccional por defecto
Supabase → Notion: automático después de cada escritura de agente. Notion → Supabase: solo por endpoint explícito. Sin sync automático bidireccional.

— Storage mínimo viable
Solo el storage que se necesita hoy: Drive (ya existe) + Supabase (metadata). R2 se agrega cuando exista el HEE.

— Schema híbrido flexible
Columnas tipadas para lo estable y frecuentemente consultado. JSONB para lo variable, lo específico de cada cliente o rubro, y lo que puede crecer.

— Multi-tenant desde el día 1
agency_id en todas las tablas. RLS habilitado. Hoy una agencia, mañana muchas sin cambiar el schema.

— Observabilidad desde el día 1
agent_logs registra cada operación de cada agente: model, tokens, cache_hit, prompt_hash, duration, status. Sin esto no hay mejora posible.

— Error handling explícito
Cada agente tiene retry logic, timeout y dead letter. Los errores no se silencian, se loggean y se notifican.

— Agnóstico de frontend
FastAPI no sabe si el frontend es Notion, una web app o un bot. Los endpoints son la interfaz pública.

3. Arquitectura de storage — corregida

Corrección aplicada vs v2
v2 proponía que los agentes rastrearan Drive en tiempo real y que R2 fuera parte del stack inicial. Ambas cosas se corrigieron. El modelo correcto está abajo.

3.1 Mapa de responsabilidades (versión final)

Google Drive
Donde el equipo trabaja hoy y seguirá trabajando. Logos, fotos, calendarios PPT, briefs, manuales de marca. No se migra nada. Los agentes acceden por file_id cuando necesitan el contenido de un archivo específico, no de forma continua.
Supabase brand_assets
El índice de todos los assets relevantes para el sistema. Guarda: file_id de Drive, URL de Drive, tipo de asset, cliente, label. Los agentes consultan esta tabla para saber qué existe — no rastrean Drive directamente.
Cloudflare R2
Fase 3+. Solo para assets que necesitan URL pública sin autenticación. Primero caso de uso: imágenes embebidas en HTML de emails del HEE.
Notion embeds
Links a Drive mostrados en Notion para que el equipo los vea sin salir del sistema. No es storage real.

3.2 Flujo correcto de registro de un asset nuevo

El equipo sube material a Drive como siempre. Para que el sistema lo 'conozca', se registra explícitamente:

1. CM sube las fotos de la nueva campaña a la carpeta del cliente en Drive (proceso normal, sin cambios)
2. CM abre el perfil del cliente en Notion y hace clic en 'Registrar asset nuevo' (un botón que llama al endpoint POST /assets/register)
3. El endpoint recibe: client_id, file_id de Drive, tipo de asset, label opcional
4. El backend llama a Drive API para verificar que el archivo existe y obtener los metadatos (nombre, mime_type, tamaño)
5. Inserta en Supabase brand_assets: {client_id, drive_file_id, drive_url, asset_type, label, mime_type, file_size_kb}
6. sync.py actualiza el perfil del cliente en Notion para mostrar el nuevo asset como embed
7. A partir de ese momento, cualquier agente puede consultar los assets del cliente vía Supabase

Por qué este flujo es más robusto
No depende de que Drive notifique cambios (que no es confiable). No requiere que la service account tenga acceso a todas las carpetas automáticamente. El registro explícito es un paso de 10 segundos para el equipo y elimina bugs silenciosos donde el agente no encuentra un archivo y falla sin aviso.

3.3 Cuándo y cómo los agentes leen Drive

Los agentes consultan Supabase para saber qué assets existen. Solo llaman a Drive API cuando necesitan el contenido real del archivo — no para saber si existe.

# Patrón correcto — agents/base_agent.py

def get_client_context(client_id: str) -> dict:
    # Los agentes siempre empiezan por Supabase
    client = supabase.table('clients').select('*').eq('id', client_id).single().execute()
    assets = supabase.table('brand_assets').select('*').eq('client_id', client_id).execute()

    # Solo si hay un PDF de manual de marca, lo descarga para incluirlo en el contexto
    manual = next((a for a in assets.data if a['asset_type'] == 'pdf_manual'), None)
    manual_text = None
    if manual:
        manual_text = drive.download_as_text(manual['drive_file_id'])  # fetch puntual

    return {
        'profile': client.data,
        'assets_index': assets.data,   # metadata de todos los assets
        'manual_text': manual_text      # contenido solo si existe y es necesario
    }


4. Sincronización Notion ↔ Supabase — modelo simplificado

Corrección aplicada vs v2
v2 proponía sync bidireccional automático. Esto genera conflictos de escritura y depende de webhooks de Notion que no son confiables. El modelo correcto es asimétrico.

4.1 El modelo correcto

Supabase → Notion (automático)
Cada vez que un agente escribe en Supabase, sync.py actualiza Notion inmediatamente después. Esta dirección es simple, no tiene conflictos y es el 95% del tráfico real.
Notion → Supabase (explícito)
Cuando Cami actualiza el perfil de un cliente en Notion (tono, donts, target), llama al endpoint POST /sync/pull/{client_id}. Este endpoint lee el estado actual de Notion y lo persiste en Supabase. No es automático, es intencional.

4.2 Implementación del sync Supabase → Notion

# notion/sync.py

def push_to_notion(entity_type: str, entity_id: str, data: dict):
    '''Llamado por los agentes después de cada escritura en Supabase.
    entity_type: 'client' | 'task' | 'campaign' | 'decision' | 'metric'
    '''
    notion_id = get_notion_id(entity_type, entity_id)
    if not notion_id:
        # El registro no existe en Notion todavía — crearlo
        notion_id = create_notion_page(entity_type, data)
        save_notion_id(entity_type, entity_id, notion_id)
        return
    # Actualizar las propiedades de la página existente
    notion.pages.update(notion_id, properties=build_notion_properties(entity_type, data))

4.3 Implementación del sync Notion → Supabase (endpoint explícito)

# api/routes/sync.py

@app.post('/sync/pull/{client_id}')
async def pull_from_notion(client_id: str):
    '''Cami llama a esto desde un botón en Notion cuando termina de editar el perfil.
    Lee el estado actual de la página del cliente en Notion y lo persiste en Supabase.
    Último escritor gana. No hay conflictos porque es una acción explícita e intencional.
    '''
    notion_id = get_notion_id('client', client_id)
    notion_data = notion.pages.retrieve(notion_id)
    parsed = parse_notion_client_page(notion_data)
    supabase.table('clients').update(parsed).eq('id', client_id).execute()
    agent_logs.write('sync', client_id, {'direction': 'notion_to_supabase', 'fields': list(parsed.keys())})
    return {'status': 'ok', 'fields_updated': list(parsed.keys())}



5. Brief como JSON schema — la base del intake_agent

El brief template del Drive tiene campos definidos. Esos campos se formalizan como un schema Pydantic. Claude usa Structured Outputs para garantizar que el intake siempre devuelve exactamente esos campos, sin variaciones.

5.1 El schema Pydantic del brief

# config/schemas/brief_schema.py
from pydantic import BaseModel, Field
from typing import Optional, List

class BriefSchema(BaseModel):
    # Identidad
    client_name:         str
    industry:            str
    country:             str = 'AR'

    # Contacto
    email:               Optional[str] = None
    whatsapp:            Optional[str] = None
    website:             Optional[str] = None
    instagram:           Optional[str] = None

    # Perfil de marca — campos críticos para los agentes
    brand_voice:         str   # Cómo habla la marca
    five_brand_words:    List[str]   # 5 palabras que definen la marca
    communication_objectives: List[str]   # mínimo 3
    target_audience:     str   # a quién le habla
    target_opportunity:  Optional[str] = None   # target potencial
    value_proposition:   str   # qué la diferencia
    competitors:         List[str] = []

    # Reglas de contenido — las más consultadas por agentes
    dos:                 List[str] = []
    donts:               List[str] = []

    # Estética
    brand_colors:        List[str] = []   # hex
    fonts:               List[str] = []
    has_brand_manual:    bool = False
    brand_manual_url:    Optional[str] = None
    aesthetic_references: List[str] = []
    content_references:  List[str] = []

    # Operacional
    stories_frequency:   Optional[str] = None
    posts_per_week:      int = 3
    most_important_event: Optional[str] = None
    client_expectations: Optional[str] = None

    # Extra — cualquier campo que no encaje arriba
    extra:               dict = {}

5.2 Cómo Claude usa este schema (Structured Outputs)

# agents/intake_agent.py
import anthropic, json
from config.schemas.brief_schema import BriefSchema

def process_brief_email(email_text: str) -> BriefSchema:
    client = anthropic.Anthropic()
    response = client.messages.create(
        model='claude-haiku-4-5-20251001',
        max_tokens=2000,
        system=open('config/prompts/intake_system.txt').read(),
        messages=[{
            'role': 'user',
            'content': f'Extraé los campos de este brief:\n\n{email_text}'
        }],
        tools=[{
            'name': 'save_brief',
            'description': 'Guarda el brief estructurado del cliente',
            'input_schema': BriefSchema.model_json_schema()
        }],
        tool_choice={'type': 'tool', 'name': 'save_brief'}
    )
    # Claude está forzado a devolver el schema exacto — sin alucinaciones
    raw = response.content[0].input
    return BriefSchema(**raw)  # Pydantic valida que todos los campos sean correctos

Por qué tool_choice forzado
Al usar tool_choice: {type: 'tool', name: 'save_brief'}, Claude está obligado a llamar exactamente esa tool con exactamente ese schema. No puede devolver texto libre ni inventar campos. Si un campo no está en el email, devuelve el default del schema. Cero alucinaciones de estructura.


6. Error handling y observabilidad — qué pasa cuando algo falla

El blueprint anterior no definía qué pasa cuando un agente falla. Esto es el componente que más duele en producción si no está definido antes de llegar ahí.

6.1 Patrón de error handling en todos los agentes

# agents/base_agent.py
import time, hashlib
from functools import wraps

MAX_RETRIES = 3
RETRY_DELAYS = [2, 5, 15]  # segundos entre reintentos

def run_with_retry(agent_fn, agent_type: str, client_id: str, **kwargs):
    prompt_hash = hashlib.md5(open(f'config/prompts/{agent_type}_system.txt','rb').read()).hexdigest()[:8]
    start = time.time()

    for attempt in range(MAX_RETRIES):
        try:
            result = agent_fn(**kwargs)
            # Éxito — loggear y retornar
            log_agent(agent_type, client_id, kwargs, result, prompt_hash,
                      int((time.time()-start)*1000), 'success')
            return result

        except anthropic.RateLimitError:
            if attempt < MAX_RETRIES - 1:
                time.sleep(RETRY_DELAYS[attempt])
                continue
            log_agent(agent_type, client_id, kwargs, None, prompt_hash,
                      int((time.time()-start)*1000), 'error', 'rate_limit_exceeded')
            notify_error(agent_type, client_id, 'rate_limit_exceeded')
            raise

        except Exception as e:
            log_agent(agent_type, client_id, kwargs, None, prompt_hash,
                      int((time.time()-start)*1000), 'error', str(e))
            notify_error(agent_type, client_id, str(e))
            raise

6.2 Notificación de errores — simple y directo

No hay Sentry, no hay PagerDuty. Para el piloto, un mail a vos cuando algo falla es suficiente y tarda 5 minutos en implementar.

# config/notifications.py
import smtplib
from email.message import EmailMessage

def notify_error(agent_type: str, client_id: str, error: str):
    msg = EmailMessage()
    msg['Subject'] = f'[Hummus Brain] Error en {agent_type}'
    msg['From'] = 'brain@agenciahummus.com.ar'
    msg['To'] = 'mateo@ejemplo.com'
    msg.set_content(f'Agente: {agent_type}\nCliente: {client_id}\nError: {error}')
    # Usar Gmail SMTP con app password — 10 minutos de setup
    with smtplib.SMTP_SSL('smtp.gmail.com', 465) as s:
        s.login(GMAIL_USER, GMAIL_APP_PASSWORD)
        s.send_message(msg)

6.3 Versionado de prompts en agent_logs

Cada ejecución de agente guarda el hash MD5 de los primeros 8 caracteres del system prompt. Cuando un prompt cambia, el hash cambia, y podés comparar el antes y el después en la tabla agent_logs.

-- Consulta útil: comparar performance entre versiones de un prompt
SELECT
  prompt_hash,
  COUNT(*) as total_calls,
  AVG(tokens_out) as avg_tokens,
  SUM(CASE WHEN status = 'error' THEN 1 ELSE 0 END) as errors,
  AVG(duration_ms) as avg_ms
FROM agent_logs
WHERE agent_type = 'brief'
GROUP BY prompt_hash
ORDER BY MIN(created_at) DESC;

-- Política base: solo ves los datos de tu agency
-- (Se replica para cada tabla)
CREATE POLICY agency_isolation ON clients
  FOR ALL
  USING (agency_id = current_setting('app.current_agency_id')::UUID);

-- En FastAPI, set antes de cada query:
def set_agency_context(supabase_client, agency_id: str):
    supabase_client.rpc('set_config', {
        'setting': 'app.current_agency_id',
        'value': agency_id
    }).execute()
